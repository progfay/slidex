# PRINCIPLE — slidex デザインシステムの思想

このファイルは「なぜそう見えるべきか」を定義する。具体的な仕様(トークン値・
レイアウト・マークアップ契約)は [DESIGN.md](./DESIGN.md) にある。迷ったら
このファイルの原則に照らして判断する。

## コンセプト

**静かな画面に、はっきりした一点。**

ダークな画面の上で、内容そのもの(言葉とコード)が主役になるスライド。
テック系の文脈で使うが、ターミナル風の記号遊びやウィンドウの模倣はしない。
装飾は仕事があるときだけ登場し、仕事がなければ存在しない。

## 原則

### 1. タイポグラフィがデザインの本体である

階層はサイズ・ウェイト・余白の差だけで作る。見出しに下線・バー・アイコン
などの装飾要素を付けない。「大きくて太い」以上の説明が要らない見出しが
正しい見出しである。

### 2. アクセントは一点集中

シアンは「今このスライドで一番見てほしいもの」にだけ使う。目安は
1スライドに1箇所。見出しの一語、コードの1トークン、図の1要素。
アクセントが3箇所あるスライドは、強調が0箇所のスライドと同じである。

### 3. 色数を増やさない。濃淡で語る

基本は「背景 / インク / ミュート / アクセント」の4役で構成する。
セマンティック色(success / warning / danger)は意味があるときだけ使い、
飾りには使わない。コードのハイライトもこのパレットの色相を再利用し、
新しい色相を持ち込まない。

### 4. 最後列から読めることが最低条件

汎用システムとして、登壇(遠距離)・画面共有(中距離)・配布(近距離)の
すべてで成立させる。ただし基準は常に一番厳しい「会場の最後列」に置く。
本文の最小サイズとコントラスト比は DESIGN.md の値を下回らない。
情報を増やしたくなったら、文字を小さくするのではなくスライドを分ける。

### 5. 生成されるコンテンツを前提に、崩れにくくする

スライドは Claude Code が生成する。人間が1枚ずつ微調整する前提を置かない。
どのレイアウトも「素直な HTML を置けばそれなりに整う」デフォルトを持ち、
テキストが1行でも3行でも破綻しないこと。凝った調整が必要なデザインは、
それ自体がこのシステムでは誤りである。

## 判断に迷ったときのガイド

- 装飾を足すか迷ったら → 足さない
- 色を足すか迷ったら → 濃淡(muted / ink の使い分け)で解決できないか先に試す
- 文字を小さくするか迷ったら → スライドを分ける
- レイアウトを新設するか迷ったら → 既存レイアウト+スライド固有 `<style>` で
  済ませられないか先に試す
